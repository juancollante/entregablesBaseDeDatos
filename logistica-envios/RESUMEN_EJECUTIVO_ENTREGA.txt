🏗️ COMPONENTES DEL MODELO
═════════════════════════════════════════════════════════════════

TABLAS PRINCIPALES (6):

  1️⃣ ESTADOS_ENVIO
     └─ Catálogo estático: CREADO, EN_TRANSITO, EN_SEDE, ENTREGADO, INCIDENCIA

  2️⃣ DIRECCIONES
     └─ Normalizada: Municipios Antioquia (Colombia), códigos DANE, referencias

  3️⃣ REMITENTES
     └─ Datos quién envía: nombre, documento, email, teléfono, dirección

  4️⃣ DESTINATARIOS
     └─ Datos quién recibe: nombre, documento, email, teléfono, dirección

  5️⃣ ENVIOS (Principal)
     └─ Guía única, estado, remitente, destinatario, peso, fechas, auditoría

  6️⃣ AUDITORIA_ENVIOS
     └─ Append-only: registra CREATE, UPDATE, DELETE, ESTADO_CAMBIO con JSONB

RELACIONES: 8 relaciones FK (1:N) + restricciones de integridad
ÍNDICES: 15 índices (básicos + especializados) para optimización
VISTAS: 8 vistas analíticas (dashboards, reportes, estadísticas)
FUNCIONES: 4 funciones reutilizables (generador, validaciones, cálculos)
PROCEDIMIENTOS: 4 procedimientos transaccionales (CRUD + auditoría)

✨ CARACTERÍSTICAS PRINCIPALES
═════════════════════════════════════════════════════════════════

✅ DISEÑO NORMALIZADO
   └─ Todas tablas en BCNF (de Boyce-Codd Normal Form)
   └─ Sin redundancia, integridad referencial garantizada
   └─ Escalable para millones de registros

✅ AUDITORÍA INTEGRADA
   └─ Tabla auditoria_envios con JSONB
   └─ Automática en procedures: QUIÉN, QUÉ, CUÁNDO
   └─ Retención legal mínimo 1 año

✅ VISTAS ANALÍTICAS
   └─ Dashboard operacional (envíos por estado)
   └─ Análisis geográfico (por municipio destino)
   └─ Métricas diarias (KPIs)
   └─ Alertas (envíos con retraso)
   └─ SLA tracking (días en cada estado)
   └─ CRM (remitentes frecuentes)
   └─ Compliance (auditoría detallada)

✅ LÓGICA EN BASE DE DATOS
   └─ Generador universal de guías (número único ECV-AAAA-NNNNNNN)
   └─ Validaciones integradas (email, DANE, peso positivo)
   └─ Transacciones atómicas con rollback automático
   └─ Cambios de estado con auditoría en 1 llamada

✅ SEGURIDAD & INTEGRIDAD
   └─ Restricciones CHECK (validaciones a nivel BD)
   └─ Constraints FK con DELETE policies (RESTRICT vs CASCADE)
   └─ Unicidad en número_guia y codigo_estado
   └─ UUIDs para escalabilidad distribuida
   └─ Referencias externas sin FK (Auth, Seguimiento)

✅ DOCUMENTACIÓN EXHAUSTIVA
   └─ 30,000+ palabras (~100 páginas equivalentes)
   └─ 15+ diagramas ASCII
   └─ 25+ ejemplos SQL listos para copiar
   └─ FAQs técnicas respondidas
   └─ Checklist de implementación

Backend Integration:
  ✅ DTO mapeado a ENVIOS (Shipment Entity)
  ✅ PartyRequest/Response → REMITENTES/DESTINATARIOS
  ✅ AddressRequest/Response → DIRECCIONES
  ✅ ShipmentResponse → vw_envios_completos
  ✅ JWT usuario_id → creado_por_usuario_id_externo
  ✅ REST endpoints (POST, PUT, GET) ↔ Procedures/Views

Funcionalidades Cubiertas:
  ✅ Registrar envío (UC1: sp_crear_envio)
  ✅ Cambiar estado (UC2: sp_cambiar_estado_envio)
  ✅ Buscar por guía (UC3: vw_envios_completos)
  ✅ Dashboard operacional (UC4: vw_envios_por_estado)
  ✅ Historial auditoría (UC5: vw_auditoria_detallada)

Restricciones Implementadas:
  ✅ peso_kg > 0 (CHECK)
  ✅ fecha_estimada >= fecha_creacion (CHECK)
  ✅ departamento = 'Antioquia' (CHECK + DEFAULT)
  ✅ pais = 'Colombia' (CHECK + DEFAULT)
  ✅ email formato válido (CHECK regex)
  ✅ numero_guia UNIQUE (previene duplicados)
